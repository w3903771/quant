# 主规划 PLAN

本文件定义两份产物的**路线图、写作约定、引用策略、协作流程**。`README.md` 管索引与进度；本文件管"怎么写、按什么标准写、按什么顺序写"。

---

## 1. 总体策略

- **两条线并行但解耦**：综述（survey）讲"是什么/为什么/全局脉络"，指南（guide）讲"怎么动手/原理推导+可运行代码"。两者章节对照（见 §5），但可独立阅读。
- **逐步执行，不求一次完成**：每个会话推进 1–3 个章节单元，写完即更新 `README.md` 进度表。跨会话续作时，先读 README 进度表 + 本 PLAN + 对应 OUTLINE。
- **质量优先于速度**：宁可单章扎实，不要全篇注水。综述每章≈一篇小综述；指南每章对齐成熟开源教程的单节结构（导入→直觉→公式→实现→实验→小结→练习）。
- **「d2l」只是编写期内部参照**（这类内部规划文档可提）；**任何会发布给读者的正式产物（网站首页/各 index/章节正文/站点名与描述）一律不出现 "d2l" 字样**，对外用中性表述（"原理+公式+可运行代码三位一体的动手教程"）。

## 2. 引用与事实核验策略（重要，防杜撰）

引用不限于论文，可含**书籍、框架官方文档、开源仓库、权威博客/课程**。但必须真实存在、可追溯。

- **🔁 动态检索贯穿全程（用户强调）**：`references.md` 与各 OUTLINE 里现在列的文献只是**起点骨架，不是上限**。写**每一章时都要现做 WebSearch / 文献检索**，按该章实际内容补充最新、最相关的资料（论文、研报、框架文档、博客、Kaggle、源码），再追加进 `references.md`。**不得把引用限定在 outline 阶段找到的那几篇**。资料搜索是持续动作，贯穿整个写作过程。

- **可凭知识直接引用（高置信、经典）**：Markowitz(1952)、Sharpe(1964) CAPM、Ross(1976) APT、Fama-French(1993/2015)、Carhart(1997)、Jegadeesh-Titman(1993)、Black-Litterman(1992)、Engle(1982) ARCH、Bollerslev(1986) GARCH、Lopez de Prado《Advances in Financial Machine Learning》(2018)、Gu-Kelly-Xiu(2020, RFS)"Empirical Asset Pricing via ML"、Grinold-Kahn《Active Portfolio Management》等；成熟框架 Qlib / FinRL / alphalens / vectorbt / backtrader / zipline / akshare / tushare / baostock。
- **必须先 WebSearch 核验再写**：所有 2023 年以后的 LLM/Agent/时序基础模型相关成果（如 BloombergGPT、FinGPT、TradingAgents、FinMem、FinAgent、TimeGPT、Lag-Llama、TimesFM、Chronos、Moirai、PIXIU/FinMA 等），核验**名称、年份、作者/机构、核心主张**后再落笔；不确定的标注 `[待核验]` 而非编造。
- **Kaggle 作为一等代码/思路来源**：金融量化的高质量、可运行、经实战检验的代码大量沉淀在 Kaggle 竞赛与公开 notebook 中。优先从相关竞赛汲取**特征工程、时序交叉验证(防泄漏)、GBDT/NN 集成、波动率建模、撮合/标签设计**等工程范式，改写为本指南的可运行示例并**注明来源竞赛/notebook 链接**。已核验竞赛见 `references.md` K 类。引用 notebook 时给作者与 URL；不照搬未核实的"魔法参数"。
- **代码 API 核验**：涉及 Qlib/akshare/FinRL 等具体 API 时，若不确定签名，标注 `[API以官方文档为准]`，避免给出看似精确实则错误的调用。
- `references.md` 维护统一编号 + 链接 + 核验状态（✅已核验 / 🟦经典可信 / ⬜待核验）。正文用 `[编号]` 或 `(作者, 年份)` 引用。

## 3. 综述写作约定

- 每章结构：**引子（这章解决什么问题）→ 原理/方法（含必要公式）→ 代表性工作与脉络 → 在 A 股/实务中的应用与坑 → 小结与延伸阅读**。
- 公式用 LaTeX（`$...$` / `$$...$$`），关键变量给中文释义。
- 重"脉络"与"取舍"：不仅罗列方法，更讲清**为什么会演化到下一步、各方法的适用边界与失败模式**。
- 适度对照中美差异（数据可得性、制度如 T+1/涨跌停/融券限制、因子有效性差异）。

## 4. 指南写作约定（对齐 d2l）

- 每节结构：**动机 → 直觉 → 数学（公式推导，适度严谨）→ 从零实现（numpy/pandas/PyTorch）→ 框架实现（Qlib/FinRL 等高层 API）→ 实验与可视化 → 小结 → 练习题**。
- 代码：**可运行**为硬指标。优先真实数据（akshare/Qlib/yfinance），并给"无网络时的合成数据兜底"。统一随机种子，标注依赖版本。
- 每章开头给"本章你将学到 / 前置知识 / 运行环境"。
- 数学与工程并重：先讲清原理再调库，避免"黑箱调包"。
- 代码风格与 d2l 一致：短 cell、可增量执行、配图说明。

### 4.1 双份代码约定（重要）
每一节同时提供**两种形态**的代码，二者必须一致、可对得上：
1. **md 内片段代码**：嵌在讲解中的最小说明性片段（teaching snippet），跟着公式/直觉走，短、聚焦单一概念。
2. **可运行完整代码**：放在 `guide/code/chXX_<slug>.py`（必要时配 `.ipynb`），从导入到出图/出指标**端到端可独立运行**，复现该节全部结果。md 中以相对路径注明对应文件（如"完整代码见 `code/ch02_meanvar.py`"）。
- 规则：md 片段是完整脚本的子集/摘录，不得出现完整脚本里不存在的"幻觉 API"。完整脚本顶部写清依赖与运行命令。
- **运行状态三态纪律（替代"每章必实跑"硬承诺，评审 R5 CRITICAL 化解）**：每个完整脚本在章顶/脚本头标注其状态，不假装跑过——
  - `已实跑 ✅`：在本仓库 uv 环境 `uv run` 通过，附关键输出（指标/图/数值），核心章（见 §6 MVP 主线）**必须**达到此态；
  - `仅静态检查 🟦`：逻辑/API 经审查但未实跑（如缺数据/缺重依赖），需说明为何未跑；
  - `环境受阻 ⬜`：因兼容/接口问题暂不可跑，记录具体原因与计划。
  - 把"双份代码一致 + 真实跑通"视为本工程区别于普通网文教程的**质量护城河**——护城河只在核心章强制，前沿章允许降级，但状态必须诚实标注。
- **Kaggle 取材**：完整脚本可改编自 Kaggle 优胜/高赞 notebook 的工程范式（如 Jane Street 的特征/CV、Optiver 的波动率特征、Trading-at-the-Close 的撮合标签），但须 ① 注明来源竞赛与 notebook、② 用 A股/通用数据重写为可独立运行、③ 剔除竞赛特定的不可复现技巧（私榜过拟合参数等），保留可迁移的原理。

### 4.3 金融知识：通俗 + 专业、详尽、与技术深度融合（0 基础起，用户重点要求）
读者可能**有 ML/Python 基础但无金融背景**。核心原则：**从 0 基础起，但金融知识要讲得既通俗又专业、越详细越好，不过度简略；并且后续每一项技术都必须"结合金融"来讲，而不是脱离金融孤立地教算法。**

1. **金融基础不是"速览框"，而是详尽模块**：
   - 设**专门的金融基础部分**（`guide/ch0F 金融市场与工具基础` 作为 Part 0 的金融地基章 + `附录D 金融概念详解词典`），系统、详尽地讲：账户/委托/撮合/做多做空/融资融券、复权/分红送转、市值与估值(PE/PB/PEG/EV)、财报三表与关键科目、指数/ETF/LOF/可转债/期货/期权/保证金、无风险利率与折现、风险与收益的金融含义、A股制度等。
   - 每个概念**先给通俗直觉(生活化类比)，再给专业定义、公式与边界条件**——两者都要，不能只留大白话。宁详勿略。
2. **每章「金融前置知识」是完整小节而非便签**：每个技术章开头用**一节**（不是 3 句话）讲清本章要用到的金融概念及其专业内涵（如 ch02 详解：收益的几何/算术之分、方差/半方差/下行风险、协方差与相关性的金融来源、分散化原理、夏普/索提诺的定义与适用边界），需要时 `[详见附录D]` 展开。
3. **技术与金融深度融合（关键）**：讲任何 ML/DL/RL/LLM 技术，都按"**这解决什么金融问题 → 方法原理 → 每个数学量/输入输出的金融含义 → 结果如何做金融解读与决策**"组织，**严禁把它写成与金融无关的通用算法教程**。例：讲 LightGBM 不止讲树与增益，更讲"特征=因子暴露、标签=未来超额收益、特征重要性=因子有效性、预测值=选股打分"。
4. **综述侧**：00 导读给"金融基础最小集"指引并指向 guide 金融地基章；术语首次出现给中文专业释义 + 一句通俗解释。
5. 写作顺序原则：**先讲清金融含义，再讲数学与代码**；金融与技术交织推进，不默认读者懂金融、也不把金融讲成"科普到此为止"。
